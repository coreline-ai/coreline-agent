## 1. Context (배경)

### 1.1 Why (왜 필요한가)
Codex CLI에서 Claude 모델을, Claude Code에서 GPT 모델을 사용하려면 API 포맷 변환 프록시가 필요. 기존 `anthropic-to-openai-proxy`는 단방향(Anthropic→OpenAI)만 지원했고, coreline-agent의 다중 provider 인프라를 활용하면 양방향 + CLI fallback까지 통합 가능.

### 1.2 Current State (구현 전)
- coreline-agent에 6개 provider (anthropic, openai, gemini, openai-compatible, codex-backend, gemini-code-assist)
- HTTP 서버 없음, CLI fallback 없음
- 각 CLI 도구는 자체 API만 호출 가능

### 1.3 Target State (구현 후)
- 9개 provider (+ claude-cli, gemini-cli, codex-cli)
- Bun.serve 기반 프록시 서버 (port 4317)
- Anthropic / OpenAI Chat / OpenAI Responses API 3개 엔드포인트
- model 이름으로 자동 provider 라우팅
- `--with-cli-fallback`으로 PATH의 CLI 바이너리 자동 등록

### 1.4 Scope Boundary
- **In scope**: 프록시 서버, CLI provider 3종, API 포맷 변환, SSE 스트리밍
- **Out of scope**: TUI 연동, v2 batch/capabilities 엔드포인트, hosted tools, human input mode

### 1.5 Reference
- [multi_model_tui](https://github.com/coreline-ai/multi_model_tui) — 프록시 설계 참고 원본

## 2. Architecture Overview

### 2.1 Design Diagram
```
Client (Codex CLI / Claude Code / curl)
  │
  ▼
Bun.serve (proxy/server.ts)
  ├── POST /anthropic/v1/messages  → mapper-anthropic.ts  → router.ts → LLMProvider.send()
  ├── POST /openai/v1/chat/completions → mapper-openai-chat.ts → router.ts → LLMProvider.send()
  ├── POST /openai/v1/responses   → mapper-openai-responses.ts → router.ts → LLMProvider.send()
  ├── GET  /health                 → provider inventory
  └── GET  /v1/providers           → provider details
                                          │
                                          ▼
                                   ProviderRegistry
                                   ├── anthropic (SDK)
                                   ├── codex-backend (OAuth)
                                   ├── gemini (SDK)
                                   ├── openai (SDK)
                                   ├── openai-compatible (fetch)
                                   ├── gemini-code-assist (OAuth)
                                   ├── claude-cli (Bun.spawn)  ← NEW
                                   ├── gemini-cli (Bun.spawn)  ← NEW
                                   └── codex-cli (Bun.spawn)   ← NEW
```

### 2.2 Key Design Decisions
| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| HTTP 프레임워크 | Bun.serve (네이티브) | 프로젝트가 Bun 런타임, 외부 deps 추가 불필요 |
| CLI provider 스트리밍 | 비스트리밍 (완료 후 일괄 emit) | 대부분의 CLI가 incremental SSE 미지원 |
| provider 선택 | model 이름 기반 heuristic | multi_model_tui의 `inferProvider()` 패턴 차용 |
| SSE 포맷 | provider별 native wire format 재현 | 클라이언트가 원본 API와 동일하게 동작 |
| 엔트리포인트 | 별도 `proxy-cli.ts` | 기존 agent TUI와 독립 실행 |

## 5. Integration & Verification

### 5.1 사용 시나리오
1. **Codex CLI → Claude**: `OPENAI_BASE_URL=http://127.0.0.1:4317/openai/v1` + model=`claude-*`
2. **Claude Code → GPT**: `ANTHROPIC_BASE_URL=http://127.0.0.1:4317/anthropic` + model=`gpt-*`
3. **CLI fallback**: `--with-cli-fallback` → PATH의 바이너리 자동 등록

### 5.2 알려진 제한
- CLI fallback은 비스트리밍 (완료 후 일괄 emit)
- Anthropic thinking/reasoning 블록은 스트리밍 시 생략
- tool 호출 왕복 시 일부 vendor-specific 필드 손실 가능

| 위험 요소 | 영향도 | 완화 방안 |
|-----------|--------|-----------|
| CLI 바이너리 미설치 | 낮음 | `which` 검사 + 자동 스킵 |
| OAuth 토큰 만료 | 중간 | codex-backend/gemini-code-assist에 이미 refresh 로직 존재 |
| 대용량 streaming 응답 | 낮음 | ReadableStream backpressure + controller.enqueue |
| model 이름 충돌 | 낮음 | exact-name → prefix → default 3단계 fallback |
