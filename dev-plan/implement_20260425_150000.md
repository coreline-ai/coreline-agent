## 1. Context (배경)

### 1.1 Why
- Wave 1-6에서 메모리 레퍼런스 **tier/self-improve/session-recall** 기반을 이식했지만, 단일 메모리 항목에 대한 **시간축/감쇠/관계/청킹/운영 지식** 계층이 부재.
- 장기 운영 에이전트가 축적한 지식을 **재활용**하려면: incident 패턴 기억, runbook 재실행, RCA 자동 제안이 필요.
- 메모리 레퍼런스가 이미 zero-dep + Markdown 기반으로 검증 완료 — **알고리즘 포팅만** 필요.

### 1.2 Current State (Wave 1-6 완료)
- **Tier system**: [src/memory/tiering.ts](../src/memory/tiering.ts), [working-set.ts](../src/memory/working-set.ts)
- **Self-improvement loop**: [src/agent/self-improve/*](../src/agent/self-improve/)
- **Session lifecycle hooks** (I1 fix 적용): [session-lifecycle-hooks.ts](../src/agent/self-improve/session-lifecycle-hooks.ts)
- **Compaction + Digest**: [compaction.ts](../src/memory/compaction.ts), [digest.ts](../src/memory/digest.ts)
- **Cross-session recall**: [session-recall.ts](../src/memory/session-recall.ts)
- **Frontmatter**: MemoryEntry에 `tier?/lastAccessed?/accessCount?/importance?` 지원, round-trip 보장

### 1.3 Target State
구현 완료 후:
- **Bitemporal**: entity 속성에 `valid_time`/`record_time` 분리 기록 + `/fact` 명령
- **Decay**: `decay_weight`/`tombstoned` 프론트매터 + soft-delete 복원
- **Links**: `[[Entity]]` forward 인덱스 + BFS N-hop 그래프
- **Chunking**: 큰 문서 자동 분할 + 정밀 검색 (LongMemEval +25pp)
- **Incident**: tool 실패 3회 연속 시 자동 에스컬레이션, symptoms/hypothesis/evidence 구조화
- **Decision**: convergence 통과 시 auto-record, incident와 양방향 링크
- **RCA**: heuristic hypothesis scoring + related incidents + suggested runbooks
- **Runbook**: 증상 패턴 매칭 + dry-run 실행 + RCA 제안

### 1.4 Scope Boundary
**In scope**:
- TypeScript 직접 포팅 (레퍼런스 구현 → TS)
- 기존 MemoryEntry/ProjectMemory 확장 (backward compat 보장)
- 8개 모듈 + 22개 슬래시 커맨드 + 1개 도구
- bun:test 단위 테스트 + E2E smoke
- 각 Wave는 독립 PR 가능

**Out of scope**:
- `health` CLI (D7: 기존 `/memory status`와 중복)
- 메모리 레퍼런스 SQLite Graph v2.0 (별도 검토)
- 메모리 레퍼런스 watch/stats/doctor/agents-hint/MCP server
- RCA LLM 전략 (D8: Wave 10+)
- Runbook 실제 실행 (D12: Wave 10+ sandboxed)
- Links backlinks (D3: Wave 10+)
- Type-aware decay multiplier / dream cycle
- Facts → working-set 자동 주입

## 6. Edge Cases & Risks

| 위험 | 영향 | 완화 |
|------|------|------|
| 기존 1045 테스트 회귀 | 높음 | Phase 0 backward compat 보장, 매 Phase `bun test` 전체 |
| Incident auto-escalation false positive | 중간 | Threshold env var override, escalation 로깅 |
| Runbook 실행 보안 | 높음 | MVP dry-run만, 실제 실행은 Wave 10+ sandboxed |
| Facts 저장소 무한 성장 | 중간 | `FACTS_MAX_ENTRIES=500` 캡, 초과 시 경고 |
| Decay rate 부동소수점 오차 | 낮음 | `round(w, 6)` 메모리 레퍼런스와 동일 |
| Link index 성능 (>1000 파일) | 중간 | Incremental scan, full rescan은 명시적 명령 |
| Chunking 청크 경계 의미 손실 | 중간 | Overlap 50 words로 완화, 사용자가 크기 조정 가능 |
| RCA heuristic false hypothesis | 중간 | User feedback loop (confirm/reject), Wave 10+ LLM |
| Decision definition 모호 | 중간 | MVP = manual + convergence auto-record, 향후 튜닝 |
| 디스크 쓰기 실패 | 높음 | 모든 write try/catch best-effort, throw 금지 |
| Bitemporal recorded_at 날짜 파싱 | 낮음 | ISO + YYYY-MM-DD 호환, 실패 시 undefined fallback |
| Incident 양방향 링크 일관성 | 중간 | decisionRecord 시 incident 파일도 자동 업데이트 |
| 동일 pattern runbook 중복 | 낮음 | Upsert semantic (merge steps, max confidence) |
| Convergence → auto-decision 과다 | 중간 | `AUTO_RECORD_DECISIONS=false` 비활성화 옵션 |
| **(v0.2 신규)** `forward.json` race condition (다중 프로세스) | 중간 | `linkScan` write 시 atomic rename(`.tmp` → 최종) + flock — Phase 3 Tasks 보강 필요 |

## 7. Execution Rules

1. **독립 모듈**: 각 Phase는 독립 PR 단위. 위 Phase Dependencies에 표시된 병렬 Phase는 동시 진행 가능.
2. **완료 조건**: 모든 태스크 체크박스 + Test Case 통과 + `bun run typecheck` 0 + `bun test` 회귀 0.
3. **테스트 실패 워크플로우**: 에러 분석 → 근본 원인 수정 → 재테스트 → 통과 후에만 다음 Phase.
4. **Phase 완료 기록**: 체크박스 체크 + 이 문서 업데이트.
5. **변경 금지 영역** ([CLAUDE.md](../CLAUDE.md) 준수):
 - 외부 신규 의존성 추가 금지 (zero new deps)
 - 기존 dev-plan 파일 수정/삭제 금지
 - `SUB_AGENT_MAX_DEPTH > 2` 금지
6. **하위 호환**: 모든 신규 frontmatter 필드 optional. tier/lastAccessed 없는 기존 메모리 → recall 기본값.
7. **Best-effort I/O**: evidence/fact/decay/link/incident/decision/runbook write 실패는 never throw.
8. **문서 동기화**: API/CLI 변경 시 `docs/memory-system.md` + `docs/memkraft-integration.md` + `CLAUDE.md` 같은 PR에서 갱신.
9. **회귀 검증**: 매 Phase 종료 시 `bun test` 전체 + smoke 통과 후 commit.
