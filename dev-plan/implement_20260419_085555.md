# implement_20260419_085555.md

작성 일시: `2026-04-19 08:55:55 KST`

이 문서는 이미 구현된 싱글 에이전트 기능 위에 **에이전트가 스스로 판단하고, 기록하고, 복구하고, 검증하는 능력**을 보강하기 위한 개발 범위와 구현 순서를 고정하는 작업 문서다.

## 개발 목적
- 현재 `coreline-agent`는 싱글 에이전트 루프, 도구 실행, permission, memory, session, transcript, autopilot, hook engine, proxy/status를 갖추고 있다.
- 다음 보강 목표는 새 기능을 크게 늘리는 것이 아니라, 에이전트 실행의 **신뢰도와 재현성**을 높이는 것이다.
- 핵심 질문은 다음 4가지다.
  - 판단: 에이전트가 작업이 정말 끝났는지, 막혔는지, 사용자 확인이 필요한지 스스로 분류할 수 있는가?
  - 기록: 왜 그 판단을 했고 어떤 도구/권한/검증을 거쳤는지 나중에 재현 가능하게 남기는가?
  - 복구: 실패나 중단 이후 어디서부터 안전하게 이어갈지 판단할 수 있는가?
  - 검증: 완료 주장을 뒷받침하는 테스트/스모크/증거를 구조화해서 남기는가?
- 이 계획은 싱글 에이전트를 더 “완벽한 작업 실행기”에 가깝게 만드는 신뢰도 레이어를 추가한다.

## 공통 진행 규칙
- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 완료 처리한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- cloud LLM API를 기본 검증 경로로 사용하지 않는다.
- 모든 신규 smoke/benchmark는 mock-first로 설계한다.
- 기존 session/plan_run/transcript 호환성을 깨지 않는다.
- 기존 전체 테스트 기준선은 시작 시점 `509 pass / 0 fail`이었고, 구현 후 최종 검증 기준은 `538 pass / 0 fail`이다.
- 병렬 구현 시 아래 workstream 소유 경로를 지키고 서로의 파일을 되돌리지 않는다.
- **source of truth 고정:** 기존 `PlanRunRecord`, `Task`, `EvaluationResult`, `TaskVerification`, `TaskRecovery`, `AutopilotDecisionRecord`, `TranscriptEntryRecord`가 원본 기록이다.
- 신규 reliability 레이어는 원본 기록을 대체하지 않고, 원본 기록에서 파생되는 **derived summary / adapter / evidence bundle**로만 동작한다.
- `CompletionDecision`, `RecoveryCheckpoint`, `ResumeAdvice`, `VerificationPack`은 새로운 실행 상태 원본이 아니라 최종 보고/복구 안내/검증 증거 요약이다.
- Phase 1~4는 Phase 0 계약 확정 이후 병렬 구현 가능하지만, `src/session/records.ts`와 실제 runtime integration은 충돌 방지를 위해 소유자를 제한한다.
- Phase 5는 병렬 구현하지 않고 순차 통합한다.
- `implement_20260419_100000.md`와 병행 시 실행 순서는 `이 문서 Phase 0 → 이 문서 Phase 1~4 병렬 → single_agent_hardening Track 1 → 이 문서 Phase 5 순차 통합`으로 고정한다.
- 단, single_agent_hardening Track 1을 먼저 착수해야 하는 경우에도 이 문서 Phase 0의 source-of-truth 계약은 선행되어야 한다.

## 리뷰 반영 고정 규칙

### Source of Truth 매핑표

| 신규 개념 | 원본 source of truth | 역할 | 금지 사항 |
|---|---|---|---|
| `AgentRunOutcome` | `PlanRunRecord.status`, `Task.status`, `PlanExecutionResult` | 실행 전체 상태를 보고용으로 정규화 | 기존 status를 대체하지 않음 |
| `CompletionDecision` | `Task.status`, `TaskVerification`, `EvaluationResult`, tool/hook/permission 결과 | 최종 응답 전 완료/부분완료/차단/확인필요 판정 | evaluator 재구현 금지 |
| `AgentTraceRecord` | tool event, permission result, hook result, evaluator/completion decision | 판단 근거를 감사 로그로 남김 | raw prompt/file content/긴 output 저장 금지 |
| `RecoveryCheckpoint` | `PlanRunRecord`, `AutopilotDecisionRecord`, task/step 결과 | 마지막 성공/실패 지점 요약 | resume 원본 저장 형식 대체 금지 |
| `ResumeAdvice` | `RecoveryCheckpoint`, latest `PlanRunRecord` | 어디서부터 이어갈지 추천 | 실행 차단/자동 재시도 강제 금지 |
| `VerificationPack` | `TaskVerification`, test/build/smoke/tool evidence | 완료 주장에 붙일 증거 묶음 | 테스트/도구를 직접 실행하는 executor 역할 금지 |

### 코드 확인 결과

- session 저장 연결 확인 완료: `src/session/storage.ts`의 `appendPlanRunRecord`, `loadPlanRuns`, `loadLatestResumablePlanRun`

### Trace 저장 제한

- trace metadata는 구조화된 짧은 문자열/상태/ID 중심으로 저장한다.
- raw prompt, API key, credential, full file content, full command output은 저장하지 않는다.
- command/tool output은 요약 또는 앞/뒤 일부만 저장하며, 구현 시 최대 길이 상수를 둔다.
- redaction helper는 API key/token/password 패턴을 기본 마스킹한다.
- trace record는 기본 transcript replay/search 출력에 섞지 않고, 별도 debug/structured record로만 다룬다.

### Completion outcome truth table

| 조건 | 판정 |
|---|---|
| abort signal 또는 aborted task 존재 | `aborted` |
| 사용자 입력/승인이 필요한 task 존재 | `needs_user` |
| permission deny 또는 blocking hook으로 실행 차단 | `blocked` |
| failed task가 있고 회복 근거 없음 | `failed` |
| 완료 task는 있으나 일부 pending/running/검증 부족 | `partial` |
| 모든 task가 completed/verified이고 핵심 검증 evidence 존재 | `completed` |
| 판정 근거가 불충분 | `unknown` |

### Recovery advisory only

- `doNotRepeat[]`는 v1에서 **권고 전용**이다.
- `doNotRepeat[]`를 근거로 tool 실행을 자동 차단하지 않는다.
- 각 `doNotRepeat` 항목에는 이유와 근거 task/tool id를 함께 둔다.
- 실제 재실행/계획수정/사용자확인은 기존 runner/autopilot/사용자 흐름이 결정한다.

## 잔여 리스크 / 후속 과제
- 너무 많은 기록을 남기면 session 파일이 커질 수 있으므로 trace는 요약/구조화/샘플링 옵션을 고려한다.
- trace metadata에 API key, file content, 긴 command output이 들어가지 않도록 redaction 테스트를 둔다.
- completion judge가 과도하게 보수적이면 매번 partial이 될 수 있으므로 evidence 기준을 테스트로 고정한다.
- recovery advice는 실제 자동 재실행이 아니라 “추천”부터 시작한다.
- file diff preview/변경 전 승인 UX는 유용하지만, 이번 계획에서는 verification pack 후속 후보로만 둔다.
- benchmark 점수화는 mock-first 기본판 이후 별도 계획에서 확장한다.
- Phase 5 통합 전까지 신규 reliability 타입은 기존 plan/session/autopilot 원본 기록을 대체하지 않는다.
- `src/session/records.ts` 변경은 WS-C에서만 수행하고, storage/resume 연결은 Phase 5에서 순차 통합한다.
