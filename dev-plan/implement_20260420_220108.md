# implement_20260420_220108.md

작성 일시: `2026-04-20 22:01:08 KST`

이 문서는 기존 프로젝트별 Memory 시스템 위에 **전역 사용자 프로필 메모리 v2**를 추가하기 위한 개발 범위를 고정하고, 구현이 기존 project memory / AGENT.md / session / auto-summary 계약을 깨지 않도록 관리하기 위한 작업 문서다.

## 개발 목적
- 프로젝트가 바뀌어도 유지되는 사용자 공통 선호, 작업 방식, 응답 스타일, 반복 지시를 안전하게 저장/로드한다.
- 기존 프로젝트별 memory는 그대로 유지하고, 전역 memory는 낮은 우선순위의 advisory context로만 동작하게 한다.
- 잘못 저장된 전역 기억이 모든 프로젝트에 오염되는 위험을 줄이기 위해 scope, provenance, 민감정보 차단, 사용자 관리 명령을 함께 설계한다.
- 현재 기준선 `722 pass / 0 fail / 125 files`를 유지하면서 memory v2를 단계적으로 도입한다.

## 문서 기반 제약 사항
- `AGENT.md` / `CLAUDE.md` 로딩은 memory 시스템과 독립이어야 한다.
- project memory 저장 경로는 계속 `~/.coreline-agent/projects/{projectId}/memory/`다.
- `MemoryWrite`는 기본적으로 확인을 요구해야 한다.
- 민감 정보는 MemoryWrite로 저장하지 않는다.
- memory는 active task state의 source of truth가 아니며, 참고 context다.

## 공통 진행 규칙
- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- global memory는 project memory보다 낮은 우선순위로만 system prompt에 반영한다.
- global memory 자동 저장은 반드시 conservative/review-gated로 구현한다.
- 민감정보로 보이는 값은 저장 전 차단하거나 명시적인 경고를 반환한다.
- 전체 검증 기준은 `bun run typecheck`, `bun run build`, `bun test`다.

## 데이터 계약 초안

```ts
export type MemoryScope = "project" | "global";

export type GlobalMemoryType =
  | "preference"      // 응답 언어, 스타일, 출력 구조 등
  | "workflow"        // 커밋 분리, dev-plan 선호, 테스트 선호 등
  | "environment"     // 자주 쓰는 런타임/CLI 선호 단, 프로젝트 규칙보다 낮음
  | "feedback"        // 반복 교정/피드백
  | "reference";      // 반복 참조용 사용자 제공 정보

export interface GlobalUserMemoryEntry {
  name: string;
  type: GlobalMemoryType;
  description: string;
  body: string;
  filePath: string;
  createdAt: string;
  updatedAt: string;
  provenance: {
    source: "manual" | "memory_tool" | "auto_summary_candidate";
    sessionId?: string;
    projectId?: string;
    cwd?: string;
  };
}

export interface MemoryLoadResult {
  globalEntries: GlobalUserMemoryEntry[];
  projectEntries: import("./types.js").MemoryEntry[];
  precedence: "prompt > project_instructions > project_memory > global_memory > defaults";
}
```

## 잔여 리스크 / 후속 과제
- global memory가 부정확하게 저장되면 모든 프로젝트에 영향을 줄 수 있다.
- project-specific preference와 global preference 충돌 UX가 복잡해질 수 있다.
- 자동 저장을 과하게 열면 memory 오염이 발생할 수 있다.
- semantic search/vector memory는 별도 v3 계획으로 분리한다.
- cloud sync/team memory는 현재 single-agent local-first 성격과 다르므로 별도 제품 판단이 필요하다.
