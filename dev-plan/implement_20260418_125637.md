# implement_20260418_125637.md

작성 일시: `2026-04-18 12:56:37 KST`

이 문서는 `coreline-agent`의 싱글 에이전트 1차 완성 이후, **검증 품질 · 복구 정책 · 작업 재개 신뢰성**을 높여 “에이전트처럼 보이는 실행기”에서 “실제로 끝까지 수행하는 싱글 에이전트”로 한 단계 더 끌어올리기 위한 후속 작업 문서다.

## 개발 목적
- Phase 1~3에서 만든 goal-mode / task-state / resume 기반 위에, **완료 판정과 복구 품질**을 강화한다.
- 자유 텍스트 위주의 성공/실패 판정을 줄이고, **결정적 검증 계약**을 task 실행 루프에 더 강하게 연결한다.
- 중단 후 재개, 반복 실패, tail replan 상황에서 **더 예측 가능한 싱글 에이전트 동작**을 만든다.
- 멀티 에이전트 확장보다 먼저, **root agent 1명 기준 완성도**를 높인다.

## 개발 범위
- task별 verification contract / verification hint 구조 추가
- deterministic verifier 우선 실행 경로 강화
- recovery policy 세분화 (`failed` / `blocked` / `needs_user`) 및 retry/replan 선택 개선
- goal resume 시 active task / 마지막 verification / recovery 문맥 복원 강화
- task 간 결과 전달을 위한 structured artifact/output shape 추가
- planner가 verification 가능한 task를 만들도록 품질 보정
- TUI/CLI goal-mode 상태 표시 최소 polish
- 관련 테스트 및 README / dev-plan 반영

## 제외 범위
- 멀티 에이전트 orchestration 확장
- remote/distributed runtime 확장
- 새로운 provider / proxy / MCP 기능 추가
- TUI 대규모 디자인 리뉴얼
- 장기 memory 알고리즘 변경
- child agent depth 추가 확장

## 공통 진행 규칙
- 이번 계획은 **싱글 에이전트 품질 고도화**에만 집중하고, 멀티 에이전트/remote 방향으로 범위를 넓히지 않는다.
- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- verification은 항상 **deterministic > structural > llm** 우선순위를 유지한다.

## 문서 기반 제약 사항
- 현재 `coreline-agent`는 이미 `goal-mode`, `task-state`, `retry/replan`, `resume`의 최소 경로를 갖고 있으므로, 이번 계획은 **새 모드 추가가 아니라 기존 loop의 판정 품질 강화**가 중심이다.
- session structured record가 active task truth source라는 원칙은 유지하고, memory는 계속 참고용으로만 사용한다.
- planner 품질 개선은 recovery/verification 기반 위에서만 진행한다. planner만 따로 똑똑해지는 방향으로 확장하지 않는다.

## 잔여 리스크 / 후속 과제
- verification contract가 과하게 복잡해지면 오히려 planner/executor가 무거워질 수 있으므로 최소 종류부터 시작해야 한다.
- artifact handoff를 너무 범용적으로 설계하면 상태 저장량이 급증할 수 있으므로, 우선 summary + verification + 핵심 artifact 정도로 제한하는 것이 안전하다.
- planner 품질 개선은 항상 state/recovery 품질보다 뒤에 와야 하며, 그 우선순위를 뒤집지 않는다.
