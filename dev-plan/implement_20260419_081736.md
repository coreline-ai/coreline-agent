작성 일시: `2026-04-19 08:17:36 KST`

이 문서는 `harness-protect` 분석 결과를 바탕으로, `coreline-agent`에 **Hook Engine 최소 런타임 v1**을 우선 도입하고, 추가로 유용한 4개 패턴(권한 matcher/parser, lifecycle 경계, 설정 provenance, smoke test 표준화)을 후속 확장 Track으로 관리하기 위한 범위와 구현 순서를 고정하는 작업 문서다.

## 개발 목적
- 현재 `coreline-agent`의 `/hook/coreline/start|stop|idle`은 상태 변경용 HTTP endpoint에 가깝다.
- 이번 개발의 목적은 상태 hook과 별개로, 내부 이벤트를 받아 조건에 맞는 hook을 실행하는 **최소 Hook Engine 런타임**을 추가하는 것이다.
- `harness-protect`의 장점인 `event → matcher/if 필터 → 실행 → HookResult 수집` 구조를 가져오되, `coreline-agent`의 권한/상태/프록시 구조에 맞게 작게 적용한다.
- Phase 0~4는 **Hook Engine v1** 완성을 목표로 한다.
- Phase 5~8은 Hook Engine 구현과 섞지 않고, `harness-protect`에서 가져올 만한 후속 개선 패턴으로 분리한다.

## 공통 진행 규칙
- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 완료 처리한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- 기존 `src/proxy/hooks.ts`의 clideck 상태 hook은 유지하고, 새 Hook Engine은 `src/hooks/*`에 분리한다.
- command hook은 최소 런타임 기본 범위에서 제외한다. 필요 시 별도 dev-plan으로 분리한다.
- 권한/permission 흐름에 연결할 때는 `permission deny > permission ask > PreTool hook blocking > allow` 우선순위를 지킨다.
- hook timeout/error/non-2xx는 기본적으로 전체 agent 동작을 중단하지 않는 **fail-open** 정책을 따른다.
- 명시적으로 `blocking: true`를 반환한 hook만 tool 실행을 차단한다.
- `SessionStart`/`SessionEnd` 실제 dispatch는 Phase 6 lifecycle 경계에서 연결한다. Phase 2에서는 `StatusChange`만 안정화한다.
- 현재 전체 테스트 기준선은 최근 검증 기준 `509 pass / 0 fail`이며, 구현 후 회귀 없이 유지한다.
- Phase 5~8은 Hook Engine 이후 확장 Phase다. Track A 구현과 섞지 않고 필요 시 별도 병렬 workstream으로 분리한다.
- Phase 4에서 Track A가 완료되면, Phase 5~8은 별도 dev-plan으로 분리할지 먼저 판단한다.

## 잔여 리스크 / 후속 과제

### 유지해야 할 안전 정책
- HTTP hook executor는 SSRF 위험이 있으므로 기본 localhost-only 정책과 allowlist 테스트를 계속 유지한다.
- PreTool hook은 permission deny/ask 이후에만 실행한다. 현재 삽입 위치는 `runToolCalls()` 내부 schema validation 이후, `tool.call()` 직전이다.
- hook timeout/error/non-2xx는 fail-open으로 유지하고, 명시적 `blocking: true`만 tool 실행을 차단한다.
- provider real smoke는 기본 테스트에서 cloud LLM/API를 호출하지 않고 명시적 opt-in으로만 실행한다.

### 후속 기능 후보
- command hook은 강력하지만 위험도가 높으므로 별도 계획에서 permission-gated 형태로 검토한다.
- hook config persistence는 이번 범위에서 제외했다. 필요 시 `hooks.yml` 또는 config layer 설계가 필요하다.
- 외부 HTTP hook 등록/삭제 API는 보안 위험이 크므로 별도 보안 검토 후 진행한다.
- `PostTool` 전면 적용은 타입/계약만 열어두었고 실제 적용은 후속 범위다.
- TUI/autopilot/provider smoke는 `docs/smoke.md`에 후속 후보로 남겼다.

### 완료된 우려 정리
- Phase 5~8은 이번 요청에서 함께 구현했으므로 별도 dev-plan으로 분리하지 않았다.
- Permission matcher/parser는 기존 permission 회귀 테스트와 함께 검증했다.
- Lifecycle 정리는 idempotent cleanup 테스트와 함께 검증했다.
- Config provenance는 기존 public loader API를 유지하면서 status variant로 추가했다.
