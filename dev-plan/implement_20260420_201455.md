작성 일시: `2026-04-20 20:14:55 KST`

이 문서는 Claude Artifacts 시스템 프롬프트에서 확인한 실용 패턴을 **coreline-agent 싱글 에이전트 operator UX v1**로 축소 적용하기 위한 개발 범위를 고정하고, 구현이 목적 밖으로 확장되지 않도록 하기 위한 작업 문서다.

## 개발 목적
- `/scaffold` 명령으로 도구/프로바이더/테스트 boilerplate를 빠르게 생성한다.
- `/set` 명령으로 현재 TUI 세션의 일부 runtime 값을 다음 turn부터 안전하게 조정한다.
- Context Snip으로 완료된 tool/turn 구간을 우선 압축하여 장시간 세션 안정성을 높인다.
- `/verify`와 자동 fork verifier로 typecheck/build/test를 background task로 검증한다.
- 이미 구현된 Parallel Agent Runtime v1과 충돌하지 않고, 검증/상태 표시/중단 제어를 같은 registry/scheduler 흐름에 맞춘다.

## 제외 범위
- parallel-dev child prompt 보강, Workstream Card, Owned Path 도구 강제는 이 문서에서 제외하고 별도 dev-plan으로 분리한다.
- 동적 Skill 로더, 외부 skill 설치 시스템, PDF/PPTX export, file watcher, 대규모 TUI redesign은 제외한다.
- Cross-project read, Artifact Registry, LLM-as-Judge, clarification question engine은 후속 후보로만 둔다.
- Parallel Agent Runtime v1의 task registry/scheduler를 전면 교체하지 않는다.
- one-shot CLI에서 background task를 암묵적으로 실행하지 않는다.

## 문서 기반 제약 사항
- 현재 기준선은 `705 pass / 0 fail / 123 files`로 본다.
- built-in skill catalog는 compact prompt 정책을 유지한다. 긴 parallel-dev 전문을 `src/skills/catalog.ts`에 직접 넣지 않는다.
- Fork Verifier는 새 background 실행 체계를 만들지 않고, 가능한 범위에서 `ParallelAgentScheduler`와 task registry를 재사용한다.
- Context Snip은 메시지 배열 index만 영구 source of truth로 삼지 않는다. v1은 content hash 또는 turn counter로 marker mismatch를 방어한다.
- Runtime Tweaks는 현재 진행 중인 turn을 바꾸지 않고 다음 turn부터 반영한다.

## 공통 진행 규칙
- Phase 0의 공유 계약과 타입 초안이 확정된 뒤 Phase 1~4를 병렬 착수할 수 있다.
- 공유 파일(`slash-commands.ts`, `repl.tsx`, `loop.ts`)은 통합 담당자가 마지막에 병합한다.
- 각 Phase는 자체 테스트 완료 후에만 완료 처리한다.
- 구현 중 발생한 이슈는 해당 Phase의 `이슈 및 수정`에 기록하고 해결한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.

| Runtime Tweaks | `src/config/runtime-tweaks.ts`, `repl.tsx`, `agentLoop` | 값이 현재 turn에 즉시 반영된다고 오해할 수 있음 |

| Fork Verifier | `fork-verifier.ts`, scheduler 연동, TUI command | 장시간 test task orphan 위험 |

## 잔여 리스크 / 후속 과제
- parallel-dev child prompt 전문 확장, Workstream Card, Owned Path enforcement는 별도 dev-plan으로 분리한다.
- `maxResultChars` runtime tweak는 tool별 `maxResultSizeChars`를 초과해서 올리지는 않고, 더 낮은 전역 상한으로만 동작한다.
- plan/goal completion hook이 명확하지 않은 경로는 자동 verifier 대신 `/verify` 수동 명령으로 시작한다.
- Context Snip은 실제 품질이 중요하므로 처음에는 conservative policy로 적용하고, aggressive compaction은 후속으로 둔다.
