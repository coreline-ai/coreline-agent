## 1. Context (배경)

### 1.1 Why
Wave 9 Phase 10 종료 시점에 production 투입 가능한 8개 모듈이 통합됐지만:
- 사용자가 새 슬래시 커맨드 결과를 **TUI에서 보지 못함** (REPL 렌더링 부재)
- 다중 프로세스/장기 실행 시 잠재 위험 (메모리 누수, race) 미검증
- Runbook과 RCA가 **dry-run/heuristic-only** 상태 → 실질적 자동 복구 미달성

이 문서는 이러한 격차를 메우는 작업을 우선순위별로 정리한다.

### 1.2 Current State
- 1221 tests pass (Wave 1-9 누적)
- 35+ 슬래시 커맨드 파싱은 가능하지만 9개 신규 action의 REPL 렌더링 부재
- `slash-commands.ts` 902줄 (700줄 권고 초과)
- session-lifecycle-hooks의 sessionState Map 무제한
- Evidence JSONL은 무한 누적 (compact 정책 부재)

**P0 완료 후**:
- 9개 신규 action 모두 TUI에서 결과 렌더링 (테이블, ID, 요약 형식)

### 2.3 Key Design Decisions

| 결정 | 선택 | 근거 |
|------|------|------|
| REPL 렌더링 위치 | 신규 `src/tui/handlers/` 디렉토리 (action별 1파일) | 902줄 repl.tsx 회피, F3과 시너지 |
| 분리 방식 (F3) | sub-router 패턴 — `src/tui/commands/{group}.ts` | D18 권장 그대로 |
| Backlinks 저장 (F4) | `memory/links/backlinks.json` 별도 파일, atomic write | forward.json과 동일 패턴 |
| flock 라이브러리 (R1) | **Bun built-in** `Bun.file.lock` 또는 단순 `.lock` 파일 + `mkdirSync` 원자성 | zero-dep 유지 |
| Runbook sandbox (F5) | Bun `spawn` + 새 working dir + tool permissions check + 5초 timeout | 외부 docker 없이 |
| RCA LLM (F6) | Anthropic SDK 직접 호출 (이미 deps 있음), `claude-haiku-4-5-20251001` 기본 | 비용 최소화 |
| SQLite (N1) | Bun built-in `bun:sqlite` (zero-dep!) | F4 backlinks와 둘 중 하나 — N1 선택 시 F4 skip |
| LRU cap (R5) | `applied-skill-registry`와 동일 패턴, cap 100 | 일관성 |
| Evidence retention (O1) | 90일 + 100MB whichever first, rotated to `.archive/{YYYY-MM}/` | 운영 권장 기본 |

| R5 LRU eviction된 session evidence 손실 | 낮음 | 100 cap 충분, 추가 알림 로깅 |
| F5 sandbox 탈출 (취약점) | 높음 | OS-level 격리 X — Bash matcher만으로 부족 시 docker 검토 |
| F6 LLM 비용 폭주 | 중간 | haiku 기본 + 명시적 opt-in env, monthly cap 검토 |
| N1 SQLite migration 어려움 | 높음 | F4 backlinks를 default로 두고 N1은 v3 |
| O1 archive 디스크 폭주 | 중간 | 100MB cap whichever-first, 사용자 모니터링 권장 |
| O2 캐시 invalidation 누락 → 데이터 stale | 높음 | writeEntry/deleteEntry/tierSet/tierTouch 모두에 invalidate |
| O3 health 출력 시간 (1000+ 메모리) | 낮음 | tierList cache (O2)와 시너지 |
