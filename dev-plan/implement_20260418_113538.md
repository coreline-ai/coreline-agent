이 문서는 `coreline-agent`를 “도구를 쓸 수 있는 코딩 어시스턴트”에서 “목표 중심 싱글 에이전트”로 끌어올리기 위한 3단계 개발 범위를 고정하고, 구현이 목적 밖으로 확장되지 않도록 하기 위한 작업 문서다.

## 개발 목적
- 현재 `coreline-agent`에 이미 들어간 tool loop / plan mode / memory / session / AgentTool 기반 위에서, **싱글 에이전트가 스스로 목표를 관리하고 끝까지 처리하는 구조**를 먼저 완성한다.
- 대화 중심 실행에서 벗어나 **GoalState / TaskState / Verify / Replan / Resume**가 있는 작업 중심 루프로 전환한다.
- 멀티 에이전트 확장보다 먼저, **한 명이 끝까지 처리하는 에이전트**의 기본기부터 제품 수준으로 정리한다.

## 개발 범위
- 싱글 에이전트 전용 작업 상태 모델을 기존 `Task` / `PlanRunRecord` 중심으로 재정의 또는 확장
- plan → execute → verify → replan 기본 루프를 task 중심 실행기로 승격
- task 단위 resume / recovery / verification 저장 구조 추가
- 실패 원인 기록, 재시도 제한, 다음 행동 제안 등 에이전트형 실행 상태 정리
- TUI/CLI에서 최소한의 task 상태 표시와 task resume 진입 경로 정리
- 관련 테스트 및 문서 반영

## 제외 범위
- 멀티 에이전트 orchestration 고도화
- remote/distributed scheduler 신규 확장
- child agent depth 추가 확장
- 새로운 provider / proxy endpoint 추가
- TUI 대규모 재디자인
- 장기 memory 알고리즘 고도화 자체

## 공통 진행 규칙
- 이번 계획은 **싱글 에이전트 완성**에만 집중하고, 멀티 에이전트/remote 방향으로 범위를 넓히지 않는다.
- 새 runtime을 추가로 만드는 대신, 기존 `agent/plan-execute/session/tui` 구조를 최대한 재사용한다.
- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.

## 문서 기반 제약 사항
- 현재 `coreline-agent`는 plan mode, memory, AgentTool, proxy, MCP가 모두 구현되어 있으므로, 이번 계획은 **새 기능 추가 경쟁**이 아니라 **실행 모델 중심 재구성**이어야 한다.
- `AgentTool`과 remote 방향은 이미 존재하지만, 이번 계획의 성공 기준은 “멀티 에이전트”가 아니라 **root agent 단독 완성도**다.
- 현재 session/resume은 메시지 복원 중심이므로, 이번 계획에서는 **task resume이 우선**이다.
- 현재 planner는 동작하지만 task state truth source가 약하므로, 이번 계획에서는 **planner보다 state machine이 상위**여야 한다.
- `src/agent/remote/*`는 이미 구현 완료됐지만, 이번 계획에서는 **single-agent core completion에 영향 주지 않는 범위로 유지**하고 확장은 하지 않는다.

## 실행 진입 규칙
- v1 싱글 에이전트 모드는 **명시 진입**만 허용한다.
- 기본 채팅/TUI 입력은 기존 assistant 모드를 유지한다.
- 아래 경로만 task-centric 실행 모드로 본다.
  - CLI: `--goal-mode` 또는 이에 준하는 명시 플래그
  - TUI: `/goal <목표>` 또는 이에 준하는 명시 명령
- 일반 채팅을 자동으로 goal/task 실행으로 승격하지 않는다.
- 이유: 기존 사용자 경험을 깨지 않고, state/resume/verify 흐름을 안정적으로 도입하기 위함이다.

## 상태 진실 원본
- active goal / task queue / verification / recovery 상태의 **truth source는 session structured record**다.
- message history는 display / replay / 대화 맥락 복원용이다.
- project memory는 장기 참고용 컨텍스트이며, **live task state의 원본이 아니다.**
- auto-summary memory는 후속 참고 자료일 뿐, active task 진행 상태를 대체하지 않는다.

## 최소 상태 모델 원칙
- 새 최상위 타입을 무한정 늘리지 않는다.
- 기본 방향은 아래와 같다.
  - 기존 `Task`를 확장해 `TaskState` 역할을 맡긴다.
  - 기존 `PlanRunRecord`를 확장하거나 `GoalRunRecord`로 대체해 active goal의 저장 원본으로 쓴다.
  - `VerificationState`, `RecoveryState`는 독립 거대 타입보다 **record field / step record** 중심으로 둔다.
- 최소 상태 집합:
  - `pending`
  - `running`
  - `blocked`
  - `needs_user`
  - `failed`
  - `verified`
  - `completed`
  - `aborted`
- `retry`, `replan`, `ask-user`, `stop`는 상태가 아니라 **recovery action**이다.

## 구현 메모
- Phase 1~3는 병렬 에이전트 2개(실행 루프 / UI-CLI) + 메인 통합으로 구현했다.
- Task 상태 모델은 기존 `Task` / `PlanRunRecord` 중심으로 확장했고, active goal의 truth source를 session structured record로 고정했다.
- goal mode는 CLI `--goal-mode`와 TUI `/goal <goal>`로 명시 진입되며, resumable goal run을 저장/복원한다.
- runner는 verification / recovery / retry budget / replan / completed-prefix resume을 지원하도록 강화했다.
- `needs_user`는 명시적 사용자 승인/입력/확인이 필요한 경우에만 진입하며, non-interactive CLI에서는 대기하지 않고 해당 상태로 종료해 deadlock을 피한다.
- `blocked`는 외부 의존성/provider/network/upstream 문제로 진행 불가한 경우에 사용하고, 일반 도구 실패/권한 오류는 `failed`로 유지한다.
- retry 정책은 `retryBudget`(동일 task의 로컬 재시도) → `maxReplansPerTask`(retry 소진 후 tail replan) 순으로 계층 분리했다.

## 최종 결과 요약
- 이번 3단계 계획의 성공 기준은 “더 많은 기능”이 아니라, **한 명의 에이전트가 목표를 받고 스스로 상태를 관리하며 검증까지 마치는 구조**를 갖추는 것이다.
- 이 문서 완료 후에야, 멀티 에이전트 / remote scheduler / distributed runtime 같은 후속 방향을 다시 평가한다.

## 잔여 리스크 / 후속 과제
- planner 품질보다 state/verification이 먼저 정리되어야 하므로, 초반에는 UX보다 내부 상태 설계가 더 중요하다.
- task-state를 너무 복잡하게 설계하면 오히려 실행기가 무거워질 수 있으므로 최소 상태 집합부터 시작해야 한다.
- verification이 약하면 “에이전트처럼 보이는 실행기”에 머물 수 있으므로, Phase 2~3에서 검증 기준을 강하게 잡아야 한다.
