## 1. Context (배경)

### 1.1 Why (왜 필요한가)
- coreline-agent는 [src/memory/project-memory.ts](../src/memory/project-memory.ts) + auto-summary + skills 카탈로그를 이미 갖췄지만, **메모리 항목이 누적될수록 시스템 프롬프트가 부풀어 토큰 비용·노이즈 증가**.
- 내장 스킬(`dev-plan`, `code-review`, `investigate`, `parallel-dev`)의 **성과를 측정할 데이터가 없음** — 데이터 기반 개선 불가능.
- Plan-Execute-Evaluate 루프가 **무의미하게 maxTurns까지 반복**되는 경우가 잦음 — 수렴 자동 종료 부재.
- 메모리 레퍼런스는 이미 zero-dependency로 동등한 문제를 해결한 검증된 설계(LongMemEval 98.0%) — **알고리즘 그대로 포팅**하면 된다.

### 1.2 Current State (현재 상태)
- **메모리**: `~/.coreline-agent/projects/{projectId}/memory/{name}.md` — YAML frontmatter(`name`, `description`, `type`만), tier 개념 없음.
- **시스템 프롬프트**: [src/agent/system-prompt.ts](../src/agent/system-prompt.ts) — 모든 ProjectMemory를 무조건 dump.
- **Auto-summary**: [src/memory/auto-summary.ts](../src/memory/auto-summary.ts) — 완료된 대화에서 단일 요약 작성, tier 분류 없음.
- **Skills**: [src/skills/registry.ts](../src/skills/registry.ts) — 호출은 되지만 결과 추적 없음.
- **Plan-Execute**: [src/agent/plan-execute/runner.ts](../src/agent/plan-execute/runner.ts) — 수렴 판정 없이 Task 완료/실패만.
- **Prompt Library**: [src/prompt/library.ts](../src/prompt/library.ts) — 저장만, evidence/eval 없음.
- **세션**: JSONL로 [src/session/store.ts](../src/session/store.ts)에 영속화되지만 **재호출/검색 메커니즘 없음**.

### 1.3 Target State (목표 상태)
구현 완료 후:
- ProjectMemory 항목이 `tier: core | recall | archival` 분류됨, 시스템 프롬프트는 **working set(core 전체 + 최근 recall N개)** 만 주입.
- 세션 시작 시 `MEMORY.md` 파일이 자동 생성/갱신되어 사용자가 git diff로 변천사 추적 가능.
- Skill·Subagent·Prompt Library 호출이 자동으로 evidence JSONL 기록 → `coreline /skill stats`로 pass_rate 표시.
- Plan-Execute 루프가 최근 2회 iteration이 stable + unclear=0 시 자동 stop.
- 30일 이상 미접근 + non-core 메모리는 `coreline /memory compact`로 archival 이동.
- 과거 세션 요약을 recall tier로 인덱싱하여 `MemoryRecall` 도구로 재호출 가능.

### 1.4 Scope Boundary (범위)
**In scope**:
- TypeScript 직접 포팅 (레퍼런스 구현 코드 → TS, 알고리즘 그대로).
- 기존 `MemoryEntry`·`buildSystemPrompt`·`auto-summary` 확장 (호환성 유지).
- 신규 모듈: `src/memory/tiering.ts`, `src/memory/working-set.ts`, `src/memory/digest.ts`, `src/memory/compaction.ts`, `src/agent/self-improve/*.ts`.
- 슬래시 커맨드 추가: `/memory compact`, `/memory stats`, `/skill stats`, `/prompt evidence`.
- 신규 도구: `MemoryRecall` (cross-session recall).
- bun:test 단위 테스트 + smoke test.

**Out of scope**:
- 임베딩/벡터 검색 (레퍼런스 구현도 stdlib only — Jaccard/Containment/IDF만).
- Python 메모리 레퍼런스를 IPC로 호출 (TypeScript 단일 프로세스 유지).
- 신규 외부 의존성 추가 (yaml은 이미 있음, 그 외 zero new deps).
- `system-prompt.md`의 운영체제 레벨 자동 학습 (사용자 명시 변경만 다룸).
- 글로벌 메모리(`~/.coreline-agent/user-memory/`)의 tier 적용 — Phase는 ProjectMemory 우선, 글로벌은 Phase 12 이후 별도 기능.
- DB/SQLite 저장소 — 레퍼런스 구현.0의 graph layer는 도입하지 않음.
- TUI 대시보드 GUI — slash command CLI만 제공.
- 기존 41개 dev-plan의 회귀 테스트 추가 (이번 변경이 깨지 않으면 충분).

### 2.2 Key Design Decisions

| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| 포팅 vs IPC | TypeScript 직접 포팅 | Bun 단일 프로세스, IPC 오버헤드 없음, 타입 안정성 |
| 신규 의존성 | 추가 없음 | 레퍼런스 구현도 zero-dep, `yaml`은 이미 사용 중 |
| 저장 포맷 | JSONL (evidence) + Markdown+YAML (memory) | 기존 [src/session/store.ts](../src/session/store.ts) 패턴 재사용 |
| 기본 tier | `recall` | 메모리 레퍼런스 tiers.py:31 호환, 기존 메모리 무영향 |
| Tier-stale 정책 | core=180d, recall=60d, archival=∞ | 메모리 레퍼런스 convergence.py:48-52 그대로 |
| Working Set 크기 | 기본 8개, 환경변수로 override | Phase 1 PoC, 데이터 쌓이면 동적 |
| Convergence window | 기본 2 (최소값) | 메모리 레퍼런스 convergence.py:192-193 |
| 평가 점수 표현 | `pass_rate ∈ [0, 100]`, `accuracy ∈ [0, 100]` | 메모리 레퍼런스 `_summarise_results` 호환 |
| 자동 승격 | access_count ≥ 3 → core | 메모리 레퍼런스에는 없으나 coreline ROI ↑, 옵트인 |
| Auto-stop 적용 위치 | runner.ts `onTaskEnd` 콜백 wrap (호출자 측) | runner.ts 직접 수정 회피, 호출자에서 evidence + gate |
| 슬래시 커맨드 패턴 | `/memory *`, `/skill *`, `/subagent *`, `/prompt *` 그룹 | V6 검증: [src/tui/slash-commands.ts](../src/tui/slash-commands.ts) `handleSlashCommand` switch에 case 추가 (별도 디렉토리 분리 안 함) |
| Tier-aware stale 정책 | core=180d / recall=60d / archival=∞ (Phase 13) | 메모리 레퍼런스 convergence.py:48-52 그대로 |
| 스킬 추적 전략 | router 진입 시점 등록 + 세션 종료 시 평가 | V2 검증: 런타임 hook 없음 → AppliedSkillRegistry로 sessionId별 LRU 추적 |
| 평가 휴리스틱 (PoC) | 정상 turn_end(`reason==="completed"`) = success, 그 외 = failure | 단순화 우선. "재질문 안 함" 등 정교화는 향후 확장 |

### 5.3 Rollback Strategy
- **부분 롤백**: 각 Phase는 git commit 단위로 분리. 문제 Phase의 commit만 revert.
- **전체 롤백**:
 - `src/memory/types.ts`의 옵셔널 필드는 삭제해도 backward-compat (frontmatter는 추가 필드 허용).
 - 신규 파일은 단순 삭제.
 - `~/.coreline-agent/projects/*/skill-evidence/`, `subagent-evidence/`, `prompt-evidence/`, `session-recall/`, `MEMORY.md` 디렉토리는 사용자가 수동 삭제 가능.
 - frontmatter에 추가된 `tier`/`lastAccessed`/`accessCount`/`importance`는 무시되어도 기존 reader가 정상 동작.
- **데이터 마이그레이션 롤백 불요** — 모든 변경이 추가형(additive) only.

## 7. Execution Rules (실행 규칙)

1. **독립 모듈**: 각 Phase는 독립적으로 구현하고 테스트한다. 위 Phase Dependencies에 표시된 병렬 가능 Phase는 동시 진행 가능.
2. **완료 조건**: 모든 태스크 체크박스 체크 + 모든 Test Case 통과 + `bun run typecheck` 0건 + `bun test` 회귀 0.
3. **테스트 실패 워크플로우**: 에러 분석 → 근본 원인 수정 → 재테스트 → 통과 후에만 다음 Phase 진행. **회피용 skip/comment 금지.**
4. **Phase 완료 기록**: 체크박스를 체크하여 이 문서에 진행 상황 기록. 같은 PR에서 이 파일도 업데이트.
5. **변경 금지 영역** ([CLAUDE.md](../CLAUDE.md) "금지 사항" 준수):
 - `node_modules/`, `dist/`, `.env` 수정 금지.
 - 기존 dev-plan 파일(`dev-plan/implement_2026*.md`) 수정/삭제 금지.
 - 외부 신규 의존성 추가 금지 (zero new deps).
 - `SUB_AGENT_MAX_DEPTH > 2` 금지.
6. **하위 호환**: 모든 변경은 추가형(additive). 기존 frontmatter 미지정 메모리는 `recall` 기본값으로 동작.
7. **best-effort I/O**: evidence/digest/recall write 실패는 throw 금지 — 에이전트 루프가 우선.
8. **문서 동기화**: API/CLI 변경 시 [docs/memory-system.md](../docs/memory-system.md) + [CLAUDE.md](../CLAUDE.md) 같은 PR에서 갱신.
9. **회귀 검증**: 매 Phase 종료 시 `bun test` 전체 실행 후 commit. 회귀 1건이라도 발견 시 다음 Phase 진행 금지.

### 8.2 병합 순서

1. **A Foundation** (Phase 0) 먼저 병합. 이후 모든 워커는 A의 타입/경로를 contract source of truth로 본다.
2. **B Memory Tier Stack** (Phase 1, 2) 병합. working set selector 시그니처 확정.
3. **C Self-Improve Stack** (Phase 5)와 **B 후속**(Phase 3, 4)은 **병렬 가능**.
4. **D Plan-Execute Integration** (Phase 10)은 C의 Phase 5 + Phase 10 자체 후 병합.
5. **E Cross-session Recall** (Phase 8)은 B의 Phase 2 후 병합.
6. **F TUI Commands** 는 각 기능 코어 함수 export 안정화 후 병합.
7. **G Docs & QA** (Phase 14)는 마지막. integration smoke 통과 시 PR ready.

### 8.3 충돌 방지 규칙

- 각 워커는 자신의 소유 경로 밖 변경을 되돌리지 않는다.
- A의 타입 변경은 모든 후속 워커에 broadcast — 단일 PR에서 처리하거나 옵셔널 필드 추가만 허용.
- C의 EvidenceRecord schema 변경은 D, E, F가 동시에 영향 → migration 도우미 (`src/agent/self-improve/migrate.ts`) 추가 후에만 변경 가능.
- `~/.coreline-agent/projects/{id}/`의 디렉토리 신설은 `src/config/paths.ts` 헬퍼를 통해서만.
- frontmatter 필드 rename 금지 (alias 추가 우선).
- MemoryRecall 도구의 input/output schema 변경 시 [docs/memkraft-integration.md](../docs/memkraft-integration.md)와 같은 PR에서 갱신.
