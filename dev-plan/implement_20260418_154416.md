# implement_20260418_154416.md

작성 일시: `2026-04-18 15:44:16 KST`

이 문서는 `coreline-agent`의 싱글 에이전트 2차 품질 강화 이후, **사용자가 계속 붙어 있지 않아도 목표 완료/막힘까지 자동으로 굴러가는 싱글 에이전트 오토파일럿 v1**의 범위를 고정하기 위한 작업 문서다.

## 개발 목적
- 이미 구현된 `goal-mode`, `verification contract`, `recovery`, `resume`, `task output handoff` 위에 **감독(supervisor) 루프**를 올려, 목표가 완료되거나 진짜로 막힐 때까지 자동 진행하게 만든다.
- `clideck`의 오토파일럿에서 좋은 점인 **계속 진행, 루프 방지, 명확한 stop policy, decision log**를 싱글 에이전트 형태로 가져온다.
- 멀티 세션 라우팅이나 child orchestration을 늘리기 전에, **root agent 1명 기준 autopilot**을 먼저 제품 수준으로 검증한다.

## 개발 범위
- 명시 진입형 싱글 에이전트 autopilot 모드 (`--autopilot`, `/autopilot <goal>` 또는 동급 UX)
- goal run을 반복 실행하는 supervisor 루프
- `completed / blocked / needs_user / aborted`를 종료 상태로 삼는 strict stop policy
- retry / replan / verify 결과를 바탕으로 다음 행동을 자동 결정하는 decision engine
- repeated failure / repeated tail / no-progress 감지를 위한 loop guard
- autopilot decision log / status summary / resume 연동
- 관련 테스트 및 README / dev-plan 반영

## 제외 범위
- 멀티 에이전트/멀티 세션 라우팅
- clideck식 browser UI / mobile remote / telemetry 콘솔
- child agent depth 추가 확장
- 새로운 provider / proxy / MCP 기능 추가
- background daemon / OS scheduler / cron 연동
- remote/distributed runtime 확장

## 문서 기반 분석
- `implement_20260418_113538.md`는 **task truth source / verify / resume**를 먼저 고정해 autopilot의 기반을 만들었다.
- `implement_20260418_125637.md`는 **verification contract / recovery policy / structured output**을 강화해, supervisor가 판단할 입력 신호를 이미 충분히 제공한다.
- 따라서 다음 단계는 planner를 더 복잡하게 만드는 것이 아니라, **지금 있는 상태와 검증 결과를 기반으로 계속 굴리는 감독 루프**를 추가하는 것이다.
- `clideck`의 장점 중 즉시 차용할 만한 것은 멀티 세션 라우팅이 아니라, **계속 진행 / loop guard / notify_user에 해당하는 strict stop policy / decision log**다.

## 최종 결과 요약
- 이번 계획의 성공 기준은 “에이전트가 더 많은 기능을 가진다”가 아니라, **사용자가 자리를 비워도 root agent 1명이 목표 완료/막힘까지 스스로 굴러가는 것**이다.
- 멀티 세션 오토파일럿은 이 문서 완료 후에야 다시 평가한다.

## 잔여 리스크 / 후속 과제
- stop policy가 너무 공격적이면 불필요하게 오래 달릴 수 있고, 너무 보수적이면 자주 멈출 수 있으므로 v1에서는 규칙을 좁게 가져가야 한다.
- loop guard는 과탐지 시 정상 진행도 막을 수 있으므로, fingerprint 기준을 최소 집합으로 시작해야 한다.
- autopilot의 성공은 planner 품질보다 verification/recovery 신호 품질에 더 크게 의존한다.
