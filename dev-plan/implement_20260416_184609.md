작성 일시: `2026-04-16 18:46:09 KST`

이 문서는 AgentTool MVP 이후 남은 v2 후속 작업의 범위를 고정하고, 구현 순서를 명확히 하기 위한 작업 문서다.

## 개발 목적
이미 구현된 AgentTool MVP(깊이 1, read-only child, 요약 결과 반환)를 기반으로, 실제 멀티 에이전트 활용도를 높이는 v2 기능을 안전한 순서로 확장한다.

## 개발 범위
- child 실행 기록 저장 및 디버그 보기
- 여러 child 동시 실행
- child별 provider/model override
- write 가능한 child + 더 강한 승인 절차
- 작업 자동 분해(coordinator) 초안

## 제외 범위
- depth > 1 재귀 실행
- remote agent / distributed scheduler
- child transcript 전체를 기본 UI에 항상 실시간 스트리밍
- 무제한 병렬 fan-out
- provider별 완전 자동 라우터 고도화

## 문서 기반 제약 사항
- v1에서 지킨 안전 원칙(depth 1, parent보다 강한 권한 금지)은 v2에서도 유지한다.
- v2 기능은 순서대로 붙이며, 앞 단계 디버그/검증 수단 없이 다음 단계를 먼저 확장하지 않는다.
- 기본 사용자 경험은 기존 AgentTool MVP를 깨지 않는 하위 호환을 유지한다.

## 공통 진행 규칙
- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- child 관련 새 기능은 session, permission, abort 회귀 테스트를 같이 추가한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.

## 개발 순서표
1. child 기록/디버그 저장
2. 여러 child 동시 실행
3. child별 모델/프로바이더 분리
4. write 가능한 child + 승인 UI
5. 작업 자동 분해(coordinator)

## 왜 이 순서로 가는가
- 1단계가 먼저 있어야 이후 병렬 실행과 write 위임에서 문제를 추적할 수 있다.
- 2단계는 v2 체감 효과가 가장 크지만, 디버그 수단 없이 먼저 붙이면 문제 원인 추적이 어려워진다.
- 3단계는 병렬 실행이 붙은 뒤에야 모델 분리의 실제 이득을 볼 수 있다.
- 4단계는 가장 위험한 범위라 앞 단계 안정화 후에만 연다.
- 5단계는 앞선 네 단계를 이용하는 상위 조합 기능이라 마지막에 간다.

## 최종 결과 요약
- v2는 "보이게 만들고 → 동시에 돌리고 → 모델을 나누고 → 쓰기까지 열고 → 자동 분해" 순서로 진행한다.
- 이 순서를 바꾸지 않는 것이 디버그 비용과 권한 리스크를 가장 낮춘다.

## 잔여 리스크 / 후속 과제
- provider별 동시성 차이 때문에 Phase 2~3에서 로컬 모델과 클라우드 모델의 정책이 달라질 수 있다.
- Phase 4의 write-capable child는 UX/보안 양쪽 리스크가 커서, 별도 수동 QA 체크리스트가 필요할 수 있다.
- Phase 5는 사실상 v3 시작점이 될 수 있으므로 범위가 커지면 다시 새 dev-plan으로 분리한다.
